# 三方审查协作工作流（Three-Party Review Workflow）

本仓库是一个协作工作区，采用「一个规划核心 + 一个执行方 + 三方审查」的模式。
所有协作都以 **GitHub 仓库（计划文件 + Pull Request）** 为桥梁，各工具之间不需要直接互联。

## 角色分工

| 角色 | 承担者 | 职责 |
|------|--------|------|
| 规划核心 | Claude Code（Ultracode） | 需求分析、总体架构、核心思路、任务拆解，产出 `plans/` 下的计划文件 |
| 执行方 | ChatGPT 桌面 app（Work/Agent 模式） | 按计划文件实现代码，推送分支并开 Pull Request |
| 审查方 A | Claude Code 深度审查（详见下节） | 多智能体交叉验证的代码审查 + 安全审查 + 科研级事实核查 |
| 审查方 B | Codex review（GitHub App，`@codex review`） | 独立代码审查 |
| 审查方 C | GPT-5.6 Pro（ChatGPT 桌面 app） | 由用户在桌面 app 中发起，对 PR diff 做第三份独立深度审查 |

## Claude 深度审查的构成

审查方 A 不是单次通读，而是按任务性质组合以下技能：

- **`/code-review`（Ultracode 多智能体模式）**：多个审查智能体按维度（正确性、边界条件、性能、可维护性）并行查找问题，每条发现再由独立的对抗性验证智能体确认或驳回，只保留经交叉验证的结论。
- **`/security-review`**：涉及输入处理、认证、数据存储、外部调用的变更，追加一轮安全专项审查。
- **`deep-research`（科研级核查）**：当项目涉及科研内容（算法出处、统计方法、领域事实、文献引用）时，用多源检索 + 对抗性事实核查验证代码所依据的方法和论断是否站得住，产出带引用的核查报告。
- **计划符合性检查**：对照 `plans/` 中对应计划文件，确认实现未偏离核心思路、未擅自扩大或缩小范围。

## 标准流程

```
需求 → [1] Claude 出计划 → [2] 执行方实现并开 PR → [3] 三方审查 → [4] 修复 → [5] 合并
```

### 1. 规划（Claude，最高推理等级）
- 用户向 Claude 描述需求。
- Claude 以**最高等级推理**（ultracode 多智能体工作流：多方案并行生成、评审打分、对抗性验证）完成规划，不做浅层单遍规划。
- 产出计划文件 `plans/NNN-<主题>.md`（格式见 `plans/TEMPLATE.md`），提交到仓库。
- 计划中「执行指令」一节是写给执行方的完整、自包含的任务说明，开头必须注明执行要求（见下）。

### 2. 执行（ChatGPT 桌面 app Work 模式 + GPT-5.6 最高推理）
- 用户把计划文件内容（或仓库链接 + 计划路径）交给执行方。
- **硬性要求：执行方必须使用 ChatGPT 桌面 app 的 Work 模式，模型选用 GPT-5.6 最高推理档位（extended/max reasoning），不得降档。**
- 执行方在新分支 `feat/NNN-<主题>` 上实现，推送并开 PR，PR 描述需引用对应计划文件。

### 3. 三方审查（并行，都落在同一个 PR 上）
- **Claude 深度审查**：用户让 Claude 审查该 PR，按上文「Claude 深度审查的构成」组合技能执行，结论落为 PR 评论。
- **Codex**：PR 上评论 `@codex review`（需先安装 Codex GitHub App 并对本仓库授权）。
- **GPT-5.6 Pro**：用户在桌面 app 中让 GPT-5.6 Pro 审查 PR diff，将结论以 PR 评论形式贴回。

### 4. 修复
- 执行方（或 Claude）根据三方审查意见修复，直至三方均无阻塞性意见。

### 5. 合并标准
- 三方审查均已完成且无未解决的阻塞性意见；
- PR 描述中的审查清单（见 PR 模板）全部勾选；
- CI（如有）通过。

## 目录约定

```
plans/           # 进行中的计划文件，NNN 递增编号，一任务一文件
  TEMPLATE.md    # 计划模板
archive/YYYY-MM/ # 每月归档：该月完成的计划 + SUMMARY.md（合并的 PR、三方审查结论要点）
WORKFLOW.md      # 本文件
.github/pull_request_template.md   # PR 三方审查清单
```

## 每月归档约定

每月 1 日（定时任务自动执行，也可随时手动触发）：
1. 把 `plans/` 中状态为「完成」的计划移入 `archive/YYYY-MM/`（上一个自然月）；
2. 汇总该月合并的 PR 与三方审查结论要点，写入 `archive/YYYY-MM/SUMMARY.md`；
3. 在 SUMMARY.md 中另加一节「本月重要决策与约定」：把当月会话中确定的工作流变更、技术决策、用户偏好提炼成要点——这是**长期记忆层**，未来任何新会话读了它就能接上上下文，旧对话即使归档或删除也不丢失关键信息；
4. 提交并推送到 `main`。该月无可归档内容则跳过，不产生空提交。

## 一次性准备事项（由用户完成）

1. 在 GitHub 上为本仓库安装并授权 **Codex GitHub App**（chatgpt.com/codex 设置中连接 GitHub），使 `@codex review` 可用。
2. 在 ChatGPT 桌面 app 中连接本仓库（GitHub 连接器 / Codex），以便 Work 模式直接读取计划文件并推送代码。
3. 之后每个任务，只需向 Claude 说需求即可，其余按上述流程流转。
